\section{Introduction}

Modern information systems are distributed across servers, networks, applications, containers, cloud resources, and security tools. Each layer produces logs, metrics, and alerts. In a Security Operations Center (SOC) context, the operational challenge is not only to collect this data, but also to transform it into understandable, prioritized, and actionable information.

The final-year internship project, officially titled \textbf{Intelligent Platform for Automated Monitoring and Security}, addresses this challenge through ARIA. The objective is to centralize security and observability events, detect threats, explain incidents with an AI-assisted layer, and automate selected corrective actions with Ansible while preserving human validation for sensitive operations. This first chapter defines the context in which ARIA was built: the host organization, the cybersecurity and SOC background, the existing situation and its limits, the problem statement, the project objectives, the proposed solution, and the methodology and planning followed during the internship.

\section{Cybersecurity and SOC Context}

A Security Operations Center is the team and tooling responsible for detecting, analyzing, and responding to security events across an organization's information system. A SOC continuously receives telemetry from many sources: host-based detection agents, network intrusion detection engines, runtime security sensors, system metrics, application logs, and manual reports. The value of a SOC depends on its ability to turn this flood of raw signals into a small number of well-understood, prioritized incidents that can be acted upon quickly.

Two operational disciplines are central to this project. The first is \textbf{monitoring and detection}, which provides visibility over the host, network, runtime, and resource layers. The second is \textbf{SOAR} (Security Orchestration, Automation, and Response), which sits above detection and coordinates investigation and remediation. SOAR aims to reduce manual effort and response time by automating repetitive analysis and corrective actions, while keeping the human analyst responsible for risky decisions. ARIA is positioned in this SOAR domain: it consumes the output of monitoring tools and adds the orchestration, intelligence, and controlled automation layer that a SOC needs.

The internship work was organized around three complementary technical fronts that together form a complete detection-to-response chain:

\begin{itemize}[leftmargin=1cm]
\item \textbf{A security monitoring foundation:} Elasticsearch, Kibana, Wazuh, Suricata, Falco, Filebeat, Telegraf, and Redis, providing visibility over the host, network, runtime, and resource layers.
\item \textbf{A cloud infrastructure:} a Huawei Cloud environment that hosts the monitoring stack and the ARIA platform in a controlled, isolated environment.
\item \textbf{A backend intelligence layer:} the ARIA platform that transforms raw Elasticsearch documents into normalized alerts, incidents, AI-assisted investigations, controlled remediation, verification, archives, and operator actions.
\end{itemize}

The complete high-level chain can be summarized as follows:

\begin{center}
\texttt{Security tools -> Elasticsearch -> Backend intelligence -> Frontend SOC operations}
\end{center}

\section{Limits of the Existing Situation}

The initial environment produced useful telemetry, but several operational limitations remained:

\begin{itemize}[leftmargin=1cm]
\item security and supervision tools produced heterogeneous data formats;
\item alerts were numerous and sometimes noisy, creating alert fatigue;
\item raw technical alerts were difficult to interpret quickly;
\item correlation between host, network, runtime, and metric events required manual effort;
\item incident explanation depended heavily on expert availability;
\item corrective actions were often manual and not consistently traceable;
\item the complete lifecycle from alert to remediation was dispersed across tools.
\end{itemize}

These limits share a single root cause: the tools deliver excellent \emph{visibility} but no shared \emph{operational language} and no \emph{controlled response} layer. Every tool speaks its own format and stops at detection, so the human analyst must do all the connecting, interpreting, deciding, and verifying by hand.

\section{Problem Statement}

The problem can therefore be formulated as follows: how can a centralized platform collect heterogeneous security and monitoring data, analyze and correlate events automatically, explain incidents through AI assistance, and automate corrective actions while keeping human control over sensitive decisions?

Several sub-problems follow from this question. First, raw alert formats differ from one source to another, which makes correlation difficult. Second, repeated or noisy alerts can hide important events. Third, the analyst needs context such as IP information, MITRE ATT\&CK techniques, source history, target host, and related incidents. Fourth, automatic remediation can be dangerous if it executes unvalidated commands or targets the wrong host. Finally, the result of a fix must be verified rather than assumed.

\section{Project Objectives}

To answer this problem, the project pursued the following objectives:

\begin{itemize}[leftmargin=1cm]
\item Build and stabilize a centralized monitoring foundation on a cloud infrastructure so that real host, network, runtime, and resource telemetry reaches Elasticsearch reliably.
\item Implement a multi-source alert pipeline that polls Elasticsearch and normalizes alerts from security and observability sources into a common model.
\item Enrich and correlate alerts using GeoIP information, MITRE ATT\&CK mappings, Sigma noise rules, deduplication, and incident grouping.
\item Provide an AI-assisted response layer able to produce investigation summaries, narratives, risk assessments, and Ansible playbooks.
\item Add approval, execution, verification, archiving, audit, and safety mechanisms around remediation workflows.
\item Expose the platform through a FastAPI backend and a Next.js dashboard for SOC analysts and administrators.
\item Allow new servers to be onboarded automatically as scoped monitored assets, and prepare the solution for production deployment.
\item Validate the implementation through automated tests, compile/build checks, API checks, and runtime verification reports.
\end{itemize}

\section{Proposed Solution: ARIA}

The proposed solution is ARIA, an intelligent SOC/SOAR platform. It connects the monitoring foundation to a backend intelligence layer and a dashboard. The platform collects events from Elasticsearch, normalizes them, filters noise, deduplicates repeated alerts, enriches indicators with GeoIP and MITRE ATT\&CK context, correlates alerts into incidents, creates investigations, generates AI-assisted explanations and Ansible playbooks, requests approval, executes controlled remediation, verifies the result, and archives the case.

\diagramimage{phase3_global_backend.png}{ARIA backend intelligence layer}

This design keeps the analyst at the center of the decision process. AI is used to accelerate understanding and propose playbooks, but the platform validates and controls execution through approval, admin-secret protection, safety checks, Ansible execution records, fix verification, and audit persistence.

The solution was built in three incremental phases that match the engineering story of this report. Phase~1 established the security monitoring foundation. Phase~2 provisioned the Huawei Cloud infrastructure that hosts it. Phase~3 implemented the ARIA backend intelligence and dashboard on top of the telemetry, and was then extended with multi-server onboarding and production-oriented hardening.

\section{Methodology and Planning}

The internship followed an Agile iterative methodology organized in two-week sprints. Each sprint delivered a usable increment, followed by a short review and planning session for the next increment. The journal of board identifies weekly work from January~12, 2026 to July~12, 2026. Early sprints focused on cloud and Linux preparation, then Elasticsearch/Kibana, Wazuh, Suricata, Falco, Filebeat, Redis, and ARIA setup. Later sprints focused on backend configuration, Sigma filtering, deduplication, AI playbook generation, watcher logic, multi-server onboarding, hardening, documentation, and final validation.

The methodology was chosen after comparing common project organization models. A strict waterfall model was not appropriate because the monitoring stack, backend workflows, and validation evidence evolved as the environment became available. A full Scrum process was also too heavy for a single-intern implementation context. The adopted approach was therefore a phased iterative method with two-week sprints: each sprint produced a usable technical base, then later sprints refined and extended it.
