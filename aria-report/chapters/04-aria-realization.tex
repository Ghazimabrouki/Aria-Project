\chapter{ARIA Functional Realization}

\section{Introduction}

The previous chapter described ARIA's internal logic. This chapter shows how that logic becomes a working product: the Next.js dashboard and the features an analyst actually uses. The platform is presented page by page; each page realizes part of the functional requirements catalogued below, working directly on data collected in Chapter~2 and transformed in Chapter~3. Each feature is explained by what it does and why, not only by its controls. Access begins at the authenticated entry point in Figure~\ref{fig:ss-login}.

The chapter is organized around the analyst workflow:

\begin{itemize}[leftmargin=1cm]
\item \textbf{Triage:} dashboard overview, alerts, incidents, and search.
\item \textbf{Investigate:} incident detail, AI-assisted investigations, runtime and infrastructure analysis, IPS map.
\item \textbf{Remediate:} approval, Ansible execution, verification, and archives.
\item \textbf{Operate and administer:} AI assistant, AI operator, assets, settings, and onboarding.
\end{itemize}

\section{Dashboard}

It loads independent widgets: quick stats, investigation statistics, trends, ARIA alert statistics, source breakdown, MITRE coverage, response metrics, and geo threats. A deliberate design choice is \emph{partial failure tolerance}: each widget loads independently, so if one data source is unavailable, the rest still render. In a SOC, one slow query should never blank the analyst's overview.

\section{Alerts}

The alerts page is the entry point to the raw signal after normalization and enrichment. It supports filtering by source, severity, status, whitelist state, MITRE technique, tactic, time window, and selected asset, and exposes alert details, evidence, an IOC panel, a whitelist action, and manual incident creation. The filter logic mirrors the pipeline: normalized into a common model, every alert uses the same controls across Wazuh, Suricata, Falco, Filebeat, and Telegraf-derived alerts. The whitelist action feeds back into enrichment, so trusting a source here suppresses future noise.

Behind the page, each alert is produced by the ingestion sequence in Figure~\ref{fig:seq-ingest}: the poller reads the per-source cursor from Redis, queries Elasticsearch, and runs every document through mappers and enrichment before upserting it into SQLite. The row the analyst sees then follows the lifecycle in Figure~\ref{fig:alert-states}; its \codepath{status} field (\texttt{active}, \texttt{processed}, \texttt{archived}) is exactly what the status filter exposes. A representative normalized record appeared in Listing~\ref{lst:normalized-alert}.

\section{Incidents}

The incidents page presents the result of correlation: related alerts grouped into a single case. Its main capabilities are:

\begin{itemize}[leftmargin=1cm]
\item list incidents and filter by status, severity, time, and asset;
\item show detail tabs with correlated alerts, timeline, and related investigations;
\item let the analyst assign the incident, change its status, or launch an investigation manually.
\end{itemize}

The page lets the analyst reason at incident level rather than over dozens of individual alerts. \section{Investigations}

The security investigation page is where a case is understood and resolved. It shows workflow stages, AI analysis (summary, narrative, risk assessment), supporting evidence, the proposed playbook, rollback information, post-rollback verification, timeline, and actions. The key realization is that available actions are \emph{driven by the backend state machine}, not the frontend: the page renders only transitions allowed by the investigation's current status. This guarantees that the interface never offers an action the backend would reject. \section{Remediation Workflow}

Remediation is not a separate page; it is a controlled sequence inside the investigation view. The analyst reviews the AI-proposed playbook, approves or edits it, satisfies the admin-secret gate, executes it through Ansible, and reads back the result and fix verification.

The interface makes every gate explicit: execution stays blocked until approval, and a case is marked resolved only after verification. This visible workflow expresses the safety boundary described in Chapter~3. Its full interaction is in Figure~\ref{fig:seq-remediation}, gate logic in Figure~\ref{fig:act-approval}, and staged execution in Figure~\ref{fig:remediation-states}.

\section{Runtime Investigations}

Runtime investigations are dedicated to Falco runtime security events and are deliberately \emph{diagnostic-first}. The runtime engine builds an evidence-based decision from Falco context, diagnostic output, findings, alert payloads, and verification history, and classifies each case. Corrective execution is possible only when the plan contains real corrective actions and explicitly marks remediation as available; container and Kubernetes cases block host-level changes until namespace and mount scope are manually validated. The page reflects this: \codepath{approve_run} appears only when the case awaits approval \emph{and} the plan has corrective actions, so diagnostic-only findings are never dressed up as executable fixes.

\begin{table}[H]
\centering
\caption{Runtime investigation decision model}
\begin{tabularx}{\textwidth}{@{}p{0.32\textwidth}Y@{}}
\toprule
\textbf{Decision} & \textbf{Meaning in the implemented platform} \\
\midrule
\codepath{no_action_expected_activity} & The event matches expected administrative or system behavior; no corrective action is proposed. \\
\codepath{observe} & Evidence does not justify a system change, but the case remains visible for recurrence monitoring. \\
\codepath{manual_review_required} & Evidence is insufficient, the context is sensitive, or the scope is container/Kubernetes-related; the analyst must review manually. \\
\codepath{remediation_not_supported_for_category} & The event category has no supported automated runtime remediation path. \\
\codepath{high_risk_action_requires_approval} & A real corrective action exists, but execution requires explicit approval and safety controls. \\
\bottomrule
\end{tabularx}
\end{table}

The classification that produces these decisions is the activity diagram of Figure~\ref{fig:act-runtime}: a Falco event is sorted by class (file integrity, process/exec, privilege, network), weighed against severity and context, and only a high or unknown case becomes a runtime investigation that runs a diagnostic playbook and an AI interpretation of the output.

\section{Infrastructure Investigations}

Infrastructure investigations apply the investigation model to resource anomalies rather than security events. When the performance subsystem detects a threshold or statistical anomaly from metric baselines, it creates a diagnostic case. The page presents the resource anomaly, gauges, a metrics snapshot, top processes, diagnostic findings, and suggested actions, with acknowledge, escalate, and archive controls. Reusing the same lifecycle lets an operator handle a runaway process with the same controlled, auditable workflow used for a security incident.

\section{Assistant}

The assistant is a contextual question-answering interface for the SOC. It provides conversations, structured sources, action confirmations, timeout handling, and asset context. Its key property is grounded answers: it scopes context to the selected asset and returns structured sources and statistics alongside the answer, so a response can be checked against the underlying data rather than taken on faith. When the LLM provider is unreachable, timeout handling keeps the interface responsive instead of hanging.

\diagramimagecompact{phase3_ai_assistant.png}{Contextual AI assistant workflow}

The grounding mechanism is the sequence in Figure~\ref{fig:seq-assistant}: the question and any focus entity go to the assistant endpoint, a context builder reads asset-scoped records from SQLite and Elasticsearch, and only that grounded context is sent to the LLM. The answer is stored with its \codepath{sources_json} so it can be checked against the underlying data.

\section{Operator}

The operator workflow is the most safety-sensitive page, because it can drive remote action from a natural-language request. It provides asset-scoped sessions, inventory host discovery, readiness banners, prompt submission, playbook preview, approval, run polling, and result display.

The operator enforces every guard in sequence, as shown in Figures~\ref{fig:ai-operator}, \ref{fig:seq-operator}, and \ref{fig:act-operator}:

\begin{enumerate}[leftmargin=1cm]
\item \textbf{Asset validation.} The selected asset must exist, be enabled, and have remediation configured. \codepath{asset_id=all}, invalid, or disabled assets are refused.
\item \textbf{Intent parsing.} The natural-language request is parsed into an intent and a playbook.
\item \textbf{Scope check.} The target is checked against the account's allowed asset scope.
\item \textbf{Risk classification.} The operation is classified as low, medium, or high risk.
\item \textbf{Approval gate.} Execution is blocked until the analyst confirms the preview; high-risk operations also require the admin secret.
\item \textbf{Re-validation.} The request is re-validated immediately before the run.
\end{enumerate}

A request never becomes an action without passing all gates.

\section{End-to-End SOC Scenarios}

The pages above are not used in isolation; they form end-to-end flows. The following scenarios show how the modules interact.

\begin{description}[leftmargin=0.6cm]
\item[Scenario 1 -- Security alert to investigation.] A Wazuh or Suricata event arrives in Elasticsearch. The poller reads the source index after the saved cursor, maps the document, removes duplicates, filters noise, enriches, persists, and broadcasts an update. Correlation links the alert to an incident, which the analyst opens and launches or reviews. The AI engine builds context and produces the explanation and proposed playbook; execution stays blocked until approval.
\item[Scenario 2 -- Runtime Falco event.] A Falco event is forwarded through Falcosidekick into the asset-specific runtime index. ARIA maps it as a runtime event, the runtime engine assigns a decision (observe, manual review, or approval-required remediation), and the frontend exposes only backend-provided actions. Diagnostic-only cases show no fake remediation button.
\item[Scenario 3 -- Infrastructure anomaly.] Telegraf metrics arrive and are read by the performance components. Redis keeps recent metrics and baselines. When a threshold or anomaly condition is detected, the infrastructure workflow creates a diagnostic investigation with resource context, top processes, findings, and acknowledge/escalate/archive actions.
\item[Scenario 4 -- Operator prompt to safe execution.] The operator selects a concrete asset and sends a natural-language request. The backend validates that the asset exists, is enabled, and has remediation configured; maps the request to a capability; generates a preview with risk information; blocks until approval; and runs through Ansible with the output recorded.
\item[Scenario 5 -- New server onboarding.] An administrator runs the monitored-server bootstrap script (Chapter~5), which installs agents, writes configurations with asset identity, verifies services and index visibility, and prints the ARIA asset payload. The administrator registers the asset, runs source validation, and then views that server's data through the asset selector.
\item[Scenario 6 -- Asset filtering prevents data mixing.] Selecting an asset sends \codepath{asset_id} to supported endpoints; the backend validates and scopes the queries. The operator refuses \codepath{asset_id=all} for execution, so remediation always targets one concrete server and unrelated servers are never mixed.
\end{description}

\section{Conclusion}

This chapter walked through the realized ARIA dashboard page by page, showing how each feature works on top of the collected telemetry and the internal logic of Chapter~3. The recurring principle is that the interface never exceeds the backend: actions are state-driven, scoping is enforced, and dangerous operations are gated. With the platform fully realized, the final technical chapter addresses automatic server onboarding and production deployment.
