It loads a set of independent widgets: quick stats, investigation statistics, trends, ARIA alert statistics, source breakdown, MITRE coverage, response metrics, and geo threats. A deliberate design choice is \emph{partial failure tolerance}: each widget loads independently, so if one data source is temporarily unavailable, the rest of the dashboard still renders. This matters in a SOC, where a single slow query should never blank out the analyst's overview.

\section{Alerts}

The alerts page is the entry point to the raw signal, after normalization and enrichment. It supports filtering by source, severity, status, whitelist state, MITRE technique, tactic, time window, and selected asset, and it exposes alert details, evidence, an indicator-of-compromise panel, a whitelist action, and manual incident creation. The logic behind the filters mirrors the pipeline: because every alert was normalized into a common model, the same filter controls work uniformly across Wazuh, Suricata, Falco, Filebeat, and Telegraf-derived alerts. The whitelist action feeds back into enrichment, so trusting a source here suppresses its future noise.

Behind the page, each alert is produced by the ingestion sequence in Figure~\ref{fig:seq-ingest}: the poller reads the per-source cursor from Redis, queries Elasticsearch, and runs every document through the mappers and enrichment before upserting it into SQLite. The row the analyst sees then follows the lifecycle in Figure~\ref{fig:alert-states}, whose \codepath{status} field (\texttt{active}, \texttt{processed}, \texttt{archived}) is exactly what the status filter on this page exposes; a representative normalized record was given in Listing~\ref{lst:normalized-alert}.

\section{Incidents}

The incidents page presents the result of correlation: related alerts grouped into a single case. It lists incidents, filters by status, severity, time, and asset, and shows detail tabs with the correlated alerts, the timeline, and related investigations. From here an analyst can assign an incident, change its status, or launch an investigation manually. The page exists so that the analyst reasons at the level of an incident, not at the level of dozens of individual alerts. \section{Investigations}

The security investigation page is where a case is understood and resolved. It shows the workflow stages, the AI analysis (summary, narrative, risk assessment), the supporting evidence, the proposed playbook, rollback information, post-rollback verification, the timeline, and a set of actions. The key realization detail is that the available actions are \emph{driven by the backend state machine}, not by the frontend: the page renders only the transitions that the investigation's current status allows. This guarantees that the user interface can never offer an action that the backend would reject. \section{Remediation Workflow}

Remediation is not a separate page but a controlled sequence that runs through the investigation view: review the AI-proposed playbook, approve or edit it, satisfy the admin-secret gate, execute through Ansible, and read back the execution result and the fix verification. The interface makes each gate explicit, so the analyst always sees that execution is blocked until approval and that a case is marked resolved only after verification. This is the visible expression of the safety boundary described in Chapter~3, whose full interaction is the end-to-end sequence of Figure~\ref{fig:seq-remediation}, whose gate logic is the approval activity of Figure~\ref{fig:act-approval}, and whose staged execution is the remediation state machine of Figure~\ref{fig:remediation-states}.

\section{Runtime Investigations}

Runtime investigations are dedicated to Falco runtime security events and are deliberately \emph{diagnostic-first}. The runtime engine builds an evidence-based decision from the Falco context, diagnostic output, findings, alert payloads, and verification history, and classifies each case. Corrective execution is possible only when the plan contains real corrective actions and explicitly marks remediation as available; container and Kubernetes cases block host-level changes until namespace and mount scope are manually validated. The page reflects this exactly: the \codepath{approve_run} action appears only when the case is awaiting approval \emph{and} the plan has corrective actions, so diagnostic-only findings are never dressed up as executable fixes.

\begin{table}[H]
\centering
\caption{Runtime investigation decision model}
\begin{tabularx}{\textwidth}{@{}p{0.32\textwidth}Y@{}}
\toprule
\textbf{Decision} & \textbf{Meaning in the implemented platform} \\
\midrule
\codepath{no_action_expected_activity} & The event matches expected administrative or system behavior; no corrective action is proposed. \\
\codepath{observe} & Evidence does not justify a system change, but the case remains visible for recurrence monitoring. \\
\codepath{manual_review_required} & Evidence is insufficient, the context is sensitive, or the scope is container/Kubernetes-related; the analyst must review manually. \\
\codepath{remediation_not_supported_for_category} & The event category has no supported automated runtime remediation path. \\
\codepath{high_risk_action_requires_approval} & A real corrective action exists, but execution requires explicit approval and safety controls. \\
\bottomrule
\end{tabularx}
\end{table}

The classification that produces these decisions is the activity diagram of Figure~\ref{fig:act-runtime}: a Falco event is sorted by class (file integrity, process/exec, privilege, network), weighed against severity and context, and only a high or unknown case becomes a runtime investigation that runs a diagnostic playbook and an AI interpretation of its output.

\section{Infrastructure Investigations}

Infrastructure investigations apply the investigation model to resource anomalies rather than security events. When the performance subsystem detects a threshold or statistical anomaly from the metric baselines, it creates a diagnostic case. The page presents the resource anomaly, gauges, a metrics snapshot, top processes, diagnostic findings, and suggested actions, with acknowledge, escalate, and archive controls. Reusing the same investigation lifecycle for infrastructure means an operator handles a runaway process with the same controlled, auditable workflow used for a security incident.

\section{Assistant}

The assistant is a contextual question-answering interface for the SOC. It provides conversations, structured sources, action confirmations, timeout handling, and asset context. Its realization detail is that answers are grounded: the assistant scopes its context to the selected asset and returns structured sources and statistics alongside the answer, so a response can be checked against the underlying data rather than taken on faith. When the LLM provider is unreachable, the timeout handling keeps the interface responsive instead of hanging.

\diagramimagecompact{phase3_ai_assistant.png}{Contextual AI assistant workflow}

The grounding mechanism is the sequence in Figure~\ref{fig:seq-assistant}: the question and any focus entity are sent to the assistant endpoint, a context builder reads the asset-scoped records from SQLite and Elasticsearch, and only that grounded context is sent to the LLM, whose answer is stored with its \codepath{sources_json} so it can be checked against the underlying data.

\section{Operator}

The operator workflow is the most safety-sensitive page, because it can drive remote action from a natural-language request. It provides asset-scoped sessions, inventory host discovery, readiness banners, prompt submission, playbook preview, approval, run polling, and result display. The logic enforces every guard in sequence: the operator validates that the selected asset exists, is enabled, and has remediation configured, refuses \codepath{asset_id=all}, invalid, or disabled assets, generates a preview or playbook with risk information, blocks execution until approval, and re-validates before the run. A request never becomes an action without passing all of these gates.

\section{Ansible Settings}

The Ansible settings configure how remediation reaches target hosts: inventory and connection parameters used by the executor. Connection tests can be run to confirm reachability. As everywhere else, credentials are handled without exposure, the inventory is written with restrictive permissions, and passwords are never placed on a command line. This page makes the remediation capability operational while keeping it within the security boundary. Listing~\ref{lst:inventory} shows a sanitized inventory entry of the form ARIA writes for an asset: the connection parameters are concrete, but the password is stored only as a secret reference (\codepath{password_secret_ref}), never as a literal value.

\section{End-to-End SOC Scenarios}

The pages above are not used in isolation; they form end-to-end flows. The following scenarios connect them so that the reader can see how the modules interact in practice.

\begin{description}[leftmargin=0.6cm]
\item[Scenario 1 -- Security alert to investigation.] A Wazuh or Suricata event arrives in Elasticsearch. The poller reads the source index after the saved cursor, maps the document, removes duplicates, filters noise, enriches, persists, and broadcasts an update. Correlation links the alert to an incident, the analyst opens it, and launches or reviews an investigation. The AI engine builds context and produces the explanation and proposed playbook; execution stays blocked until approval.
\item[Scenario 2 -- Runtime Falco event.] A Falco event is forwarded through Falcosidekick into the asset-specific runtime index. ARIA maps it as a runtime event, the runtime engine assigns a decision (observe, manual review, or approval-required remediation), and the frontend exposes only the backend-provided actions. Diagnostic-only cases show no fake remediation button.
\item[Scenario 3 -- Infrastructure anomaly.] Telegraf metrics arrive and are read by the performance components. Redis keeps recent metrics and baselines. When a threshold or anomaly condition is detected, the infrastructure workflow creates a diagnostic investigation with resource context, top processes, findings, and acknowledge/escalate/archive actions.
\item[Scenario 4 -- Operator prompt to safe execution.] The operator selects a concrete asset and sends a natural-language request. The backend validates that the asset exists, is enabled, and has remediation configured, maps the request to a capability, generates a preview with risk information, blocks until approval, and runs through Ansible with the output recorded.
\item[Scenario 5 -- New server onboarding.] An administrator runs the monitored-server bootstrap script (Chapter~5), which installs agents, writes configurations with asset identity, verifies services and index visibility, and prints the ARIA asset payload. The administrator registers the asset, runs source validation, and then views that server's data through the asset selector.
\item[Scenario 6 -- Asset filtering prevents data mixing.] Selecting an asset sends \codepath{asset_id} to supported endpoints; the backend validates and scopes the queries. The operator refuses \codepath{asset_id=all} for execution, so remediation always targets one concrete server and unrelated servers are never mixed.
\end{description}

\section{Conclusion}

This chapter walked through the realized ARIA dashboard page by page, showing how each feature works on top of the collected telemetry and the internal logic of Chapter~3. The recurring principle is that the interface never exceeds the backend: actions are state-driven, scoping is enforced, and dangerous operations are gated. With the platform fully realized for a monitored environment, the final technical chapter addresses how new servers are onboarded automatically and how the whole solution is deployed in production.
