\chapter{Multi-Server Onboarding and Production Deployment}

\section{Introduction}

So far the solution has been described as a single central node: one server hosting the monitoring stack and ARIA, observing itself. A real SOC, however, must watch many servers. This final technical chapter closes the engineering story by explaining how a new server is onboarded automatically, that is, prepared to send its own logs, security events, runtime events, and metrics to the central stack, and how it becomes a scoped asset inside ARIA. It then describes how the complete solution is deployed and hardened for production. The central instrument is a single bootstrap script that turns a bare Linux server into a fully monitored asset and prints exactly the information ARIA needs to register it.

\section{Why Multi-Server Monitoring Is Needed}

A SOC that can only observe its own host is a demonstration, not an operational tool. The value of ARIA grows with the number of servers it can see, but adding servers introduces two hard requirements:

\begin{itemize}[leftmargin=1cm]
\item \textbf{Consistent configuration.} Every new server needs the same agents, forwarding targets, and predictable index naming. Otherwise data fails to arrive or cannot be attributed to the right server.
\item \textbf{Strict separation.} Once several servers share one Elasticsearch cluster, alerts, investigations, and remediation actions for one server must never be confused with another.
\end{itemize}

The bootstrap script below solves the first requirement; the asset-scoping logic of Chapter~3 solves the second. This chapter connects the two.

\section{Monitored Server Bootstrap Script}

The onboarding script, \sourcefile{bootstrap_monitored_vm.sh}, is designed to run on a server that will be \emph{observed} by ARIA, never on the central host. Its first action is therefore a safety guard: it detects central indicators (a local Elasticsearch, Kibana, or Wazuh Manager, or a central-looking hostname) and refuses to continue unless the operator confirms with an explicit phrase. This prevents the most damaging mistake, which would be reconfiguring the central server as if it were a monitored client.

After the guard, the script collects the onboarding inputs: the server name, an environment label (such as \codepath{safe}, \codepath{prod}, or \codepath{dmz}), the server IP, the central Elasticsearch URL or IP and credentials, an optional certificate authority path, and optional Wazuh Manager details. Component flags (\codepath{--wazuh}, \codepath{--falco}, \codepath{--telegraf}, \codepath{--filebeat}, \codepath{--suricata}) let the operator install only what a given server needs, and a \codepath{--yes} mode supports non-interactive runs. The script then tests Elasticsearch reachability before installing anything, installs the selected agents, writes their configurations with the server's identity, starts and verifies the services, checks that the expected indices appear, and finally prints a ready-to-use ARIA asset payload. A sandboxed test mode (\codepath{ARIA_TEST_MODE}) allows the whole flow to be exercised safely against a fake root before touching a real server.

\section{Wazuh Agent Configuration on Monitored Servers}

On a monitored server, the script installs the \textbf{Wazuh Agent} (not the manager) and points it at the central Wazuh Manager. The manager host defaults to the Elasticsearch host unless overridden, and the agent enrolls into a configurable group through the enrollment channel on port~1515, then reports security events on port~1514. A useful detail is that the script can query the Wazuh API on port~55000 to retrieve the agent's identifier by name, so that the resulting ARIA asset records the exact \codepath{agent_id}. From then on, that server's host security alerts flow through the manager into the shared \codepath{wazuh-alerts-*} index, where ARIA can attribute them to the asset using the recorded agent identity.

\section{Suricata Configuration}

If network monitoring is requested, the script installs and configures Suricata on the monitored server using the same logic as the central node: it detects the active interface so capture happens on the right link, and it enables the EVE JSON output. On monitored servers, however, the Suricata events are shipped by the local Filebeat into the server's own \codepath{filebeat-<server>-*} index rather than a central one. This keeps each server's network events attributable, and ARIA's asset configuration records that the Suricata source for this server lives in the Filebeat index family.

\section{Filebeat Forwarding Logic}

Filebeat is the workhorse of monitored-server onboarding. The script installs it from the Elastic repository, trusts the central certificate authority, and configures it to forward the server's system and authentication logs, and its Suricata EVE events, into a per-server index named \codepath{filebeat-<server>-*}. The per-server naming is the heart of attribution: because the index name embeds the server identity, ARIA can later scope all log and network data to exactly one asset simply by targeting that pattern. Without this convention, multiple servers would write into a shared index and could not be cleanly separated.

\section{Telegraf Metrics Forwarding}

When metrics are requested, the script installs \textbf{Telegraf} from the InfluxData repository, trusts the Elasticsearch certificate authority, and configures the Elasticsearch output to write into \codepath{telegraf-<server>-*}. As on the central node, it verifies the Elasticsearch version before writing the output so the metrics pipeline is not silently broken, and it collects the standard host metrics (CPU, memory, disk, network, processes, sockets). These per-server metric indices feed the metrics page and the infrastructure investigations for that specific asset.

\section{Falco and Falcosidekick Runtime Forwarding}

For runtime monitoring, the script installs \textbf{Falco} with the modern eBPF driver and \textbf{Falcosidekick} as the forwarder, configured to send runtime events to the central Elasticsearch under \codepath{falco-<server>-*}. The same validation discipline applies: the Elasticsearch endpoint is checked before Falcosidekick is started, so a misconfigured credential or unreachable target fails loudly during onboarding rather than silently producing no runtime data. ARIA's runtime investigation engine then consumes this per-server index for the asset.

\section{Elasticsearch Target Configuration}

Every agent on a monitored server points back to the same central Elasticsearch, reached over HTTPS using the central credentials and certificate authority. The script tests this target before installing anything, which is the single most important precondition: if the server cannot reach and authenticate to Elasticsearch, no amount of agent installation will produce visible data. Centralizing the target also means a monitored server holds no analytical state of its own; it is purely a producer, and all correlation and intelligence happen centrally in ARIA.

\section{ARIA Asset Creation from Script Output}

The bootstrap script ends by printing a JSON payload for ARIA's asset API (\codepath{POST /api/v1/assets}). This is the connective tissue between onboarding and the platform: the script already knows the server's identity and exactly which sources were installed, so it emits a payload that describes them precisely. The payload carries the asset identifier, name, hostname, IP, environment, an enabled flag, a remediation flag (off by default for safety), and a \codepath{source_config_json} that lists, for each installed source, its per-server index pattern and host identity, including the Wazuh \codepath{agent_id} when applicable. The default \codepath{remediation_enabled: false} embodies a safe-by-default policy: a freshly onboarded server is observed first and only later, deliberately, allowed to be a remediation target.

\section{Containerized and Global Deployment}

Beyond individual agents, the platform itself is deployed as cooperating services. The production-oriented layout runs the HTTP API and the background workers as separate processes, so that a crash in a long-running polling or correlation loop cannot take down the API that serves the dashboard. The FastAPI application listens on port~8001 and the production dashboard is exposed on port~3001 (the Next.js container listens internally on port~3000), while Redis and SQLite are local data services and Elasticsearch is the external source shared by all monitored servers. This process separation, combined with the restart-and-graceful-shutdown handling of the background process, is what lets the platform run unattended.

\section{Production Deployment Workflow}

Putting the whole solution into production follows the same order as the report itself, which is also the order of dependencies. The cloud infrastructure and hardened central host are provisioned first; the monitoring foundation is installed and verified to deliver data into Elasticsearch; the ARIA backend and dashboard are deployed on top; monitored servers are onboarded one by one with the bootstrap script and registered as assets; each asset is source-validated; and only then is remediation selectively enabled per asset. This staged rollout means the platform is never trusted with action before it has been proven to see data correctly. Figure~\ref{fig:seq-rollout} shows the deployment steps as a sequence: setting the masked \codepath{.env}, running the SQLite migrations, enabling the backend and worker processes under \codepath{systemd}, building and starting the dashboard, running health checks, and finally applying the production hardening.

\section{Production Limitations and Hardening}

The platform was validated for an internal, trusted-network deployment, and this scope must be stated honestly. Several items must be addressed before public exposure. There is no full public authentication layer yet: the platform relies on internal trusted-network deployment and the admin-secret gate for dangerous endpoints, and role-based access control is not implemented. SQLite is appropriate for the current single-node SOC, but a higher-concurrency production deployment should strengthen the persistence layer accordingly. TLS and certificate handling must be adapted to the production environment rather than bypassed. Finally, Elasticsearch and Redis must remain internal, administrative access should be restricted to trusted networks or a VPN, and only strictly required ports should be exposed. These constraints define the production-hardening work and align with the future work in the general conclusion.

\section{Conclusion}

This chapter completed the engineering story: a single bootstrap script turns any Linux server into a consistently configured, fully attributable monitored asset, ARIA registers and source-validates that asset, and the whole platform is deployed as separated services with a staged, gated production rollout. The solution therefore scales from one self-monitored host to a fleet of servers without losing the separation and control that make it safe. The honest production limitations, authentication, RBAC, persistence, and exposure, frame the work that remains, which the general conclusion now summarizes alongside the project's overall results.
