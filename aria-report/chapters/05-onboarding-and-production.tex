\chapter{Multi-Server Onboarding and Production Deployment}

\section{Introduction}

So far the solution has been described as a single central node: one server hosting the monitoring stack and ARIA, observing itself. A real SOC, however, must watch many servers. This final technical chapter explains how a new server is onboarded to send its logs, security and runtime events, and metrics to the central stack, becoming a scoped ARIA asset. It then describes production deployment and hardening. The central instrument is a single bootstrap script that turns a bare Linux server into a fully monitored asset and prints exactly the information ARIA needs to register it.

\section{Why Multi-Server Monitoring Is Needed}

A SOC that only observes its own host is a demonstration, not an operational tool. ARIA's value grows with the servers it monitors, but adding servers introduces two requirements:

\begin{itemize}[leftmargin=1cm]
\item \textbf{Consistent configuration.} Every new server needs the same agents, forwarding targets, and predictable index naming; otherwise data fails to arrive or cannot be attributed to the right server.
\item \textbf{Strict separation.} Once several servers share one Elasticsearch cluster, alerts, investigations, and remediation actions for one server must never be confused with another.
\end{itemize}

The bootstrap script below solves the first requirement; the asset-scoping logic of Chapter~3 solves the second. This chapter connects the two.

\section{Monitored Server Bootstrap Script}

The onboarding script, \sourcefile{bootstrap_monitored_vm.sh}, runs on servers to be \emph{observed} by ARIA, not the central host. Its first action is a safety guard: it detects central indicators (local Elasticsearch, Kibana, or Wazuh Manager, or a central-looking hostname) and refuses to continue unless the operator confirms with an explicit phrase. This prevents the most damaging mistake---reconfiguring the central server as a monitored client.

After the guard, the script collects onboarding inputs: server name, environment label (such as \codepath{safe}, \codepath{prod}, or \codepath{dmz}), server IP, central Elasticsearch URL or IP and credentials, optional certificate authority path, and optional Wazuh Manager details. Component flags (\codepath{--wazuh}, \codepath{--falco}, \codepath{--telegraf}, \codepath{--filebeat}, \codepath{--suricata}) install only needed components, and a \codepath{--yes} mode supports non-interactive runs. It tests Elasticsearch reachability before installing anything, installs selected agents, writes configurations with the server's identity, starts and verifies services, checks expected indices appear, and prints a ready-to-use ARIA asset payload. A sandboxed test mode (\codepath{ARIA_TEST_MODE}) lets the whole flow be exercised safely against a fake root before touching a real server.

\section{Wazuh Agent Configuration on Monitored Servers}

On monitored servers, it installs the \textbf{Wazuh Agent} (not the manager) and points it at the central Wazuh Manager. The manager host defaults to the Elasticsearch host unless overridden, and the agent enrolls into a configurable group through the enrollment channel on port~1515 and reports security events on port~1514. The script can also query the Wazuh API on port~55000 to retrieve the agent's identifier by name, so the resulting ARIA asset records the exact \codepath{agent_id}. That server's host security alerts then flow through the manager into the shared \codepath{wazuh-alerts-*} index, where ARIA attributes them to the asset using the recorded agent identity.

\section{Suricata Configuration}

If network monitoring is requested, the script installs and configures Suricata using the same logic as the central node: it detects the active interface for capture and enables the EVE JSON output. On monitored servers, however, local Filebeat ships Suricata events into the server's own \codepath{filebeat-<server>-*} index rather than a central one. This keeps each server's network events attributable; ARIA's asset configuration records that the Suricata source for this server lives in the Filebeat index family.

\section{Filebeat Forwarding Logic}

Filebeat is the workhorse of monitored-server onboarding. The script installs it from the Elastic repository, trusts the central certificate authority, and configures it to forward the server's system and authentication logs and Suricata EVE events into a per-server index named \codepath{filebeat-<server>-*}. Per-server naming is the heart of attribution: the index name embeds the server identity, so ARIA can scope all log and network data to exactly one asset by targeting that pattern. Without this convention, multiple servers would write into a shared index and could not be cleanly separated.

\section{Telegraf Metrics Forwarding}

When metrics are requested, it installs \textbf{Telegraf} from the InfluxData repository, trusts the Elasticsearch certificate authority, and configures the Elasticsearch output to write into \codepath{telegraf-<server>-*}. As on the central node, it verifies the Elasticsearch version before writing output so the metrics pipeline is not silently broken, and collects standard host metrics (CPU, memory, disk, network, processes, sockets). These per-server metric indices feed the metrics page and infrastructure investigations for that asset.

\section{Falco and Falcosidekick Runtime Forwarding}

For runtime monitoring, it installs \textbf{Falco} with the modern eBPF driver and \textbf{Falcosidekick} as the forwarder, configured to send runtime events to the central Elasticsearch under \codepath{falco-<server>-*}. The same validation discipline applies: the Elasticsearch endpoint is checked before Falcosidekick starts, so a misconfigured credential or unreachable target fails loudly during onboarding rather than silently producing no runtime data. ARIA's runtime investigation engine consumes this per-server index for the asset.

\section{Elasticsearch Target Configuration}

Every agent points back to the same central Elasticsearch, reached over HTTPS with the central credentials and certificate authority. The script tests this target before installing anything---the single most important precondition: if the server cannot reach and authenticate to Elasticsearch, no amount of agent installation will produce visible data. Centralizing the target also means a monitored server holds no analytical state; it is purely a producer, and all correlation and intelligence happen centrally in ARIA.

\section{ARIA Asset Creation from Script Output}

The bootstrap script prints a JSON payload for ARIA's asset API (\codepath{POST /api/v1/assets}). This connects onboarding to the platform: the script already knows the server's identity and exactly which sources were installed, so it emits a payload that describes them precisely. The payload carries asset identifier, name, hostname, IP, environment, enabled and remediation flags (the latter off by default for safety), and a \codepath{source_config_json} listing each installed source's per-server index pattern and host identity, including the Wazuh \codepath{agent_id} when applicable. The script performs the following actions in order:

\begin{enumerate}[leftmargin=1cm]
\item \textbf{Environment validation.} It checks that the host is Linux, the script runs as root, systemd is available, and both Docker and Docker Compose plugin are installed.
\item \textbf{Destructive confirmation.} It requires the operator to type an explicit phrase so the script is never run accidentally on the wrong host.
\item \textbf{Central tools runner.} It executes \sourcefile{setup_script_telegraf.sh}, which installs and hardens Elasticsearch, Kibana, Filebeat, Suricata, Wazuh Manager, Falco, Falcosidekick, Telegraf, and Fail2Ban.
\item \textbf{Native service validation.} It verifies that every required service is active before continuing.
\item \textbf{\codepath{.env} validation.} It checks that a readable \codepath{.env} exists beside the Compose file, never reading or printing its contents.
\item \textbf{Compose deployment.} It runs \codepath{docker compose pull} and \codepath{docker compose up -d} from \sourcefile{aria-application/docker-compose.yml}.
\item \textbf{Container validation.} It confirms that redis, api, worker, and frontend containers are running and that Redis responds to \codepath{PING}.
\item \textbf{Health checks.} It polls the API health endpoint at \codepath{http://127.0.0.1:8001/health} and checks that the frontend responds on port~3001.
\end{enumerate}

If any layer fails, the script prints diagnostic commands and exits, leaving no half-validated platform. This makes brain-server installation a single, reproducible command rather than a fragile manual procedure.

\section{Containerized and Global Deployment}

\raisebox{-0.22cm}{\includegraphics[width=1.0cm]{assets/logos/docker.png}}\hspace{0.15cm}Beyond individual agents, the platform is deployed as cooperating services. The production-oriented layout runs the HTTP API and background workers as separate processes, so a crash in a long-running polling or correlation loop cannot take down the API serving the dashboard. The FastAPI application listens on port~8001 and the production dashboard is exposed on port~3001 (the Next.js container listens internally on port~3000), while Redis and SQLite are local data services and Elasticsearch is the external source shared by monitored servers. This process separation, combined with restart-and-graceful-shutdown handling of the background process, lets the platform run unattended.

The ARIA containers are pre-built and published to Docker Hub under the \codepath{ghaziiii/aria_project} namespace. The brain server therefore does not build images locally; it pulls the published tags:

\begin{itemize}[leftmargin=1cm]
\item \textbf{\codepath{ghaziiii/aria_project:redis-latest}} --- Redis cache and cursor store.
\item \textbf{\codepath{ghaziiii/aria_project:backend-latest}} --- FastAPI service.
\item \textbf{\codepath{ghaziiii/aria_project:worker-latest}} --- Background polling and correlation worker.
\item \textbf{\codepath{ghaziiii/aria_project:frontend-latest}} --- Next.js dashboard.
\end{itemize}

The backend and worker share the same image but run different commands: the backend starts \codepath{uvicorn} on port~8001, while the worker runs \codepath{python3 main.py} with \codepath{SKIP_RESPONSE_API=1} to avoid binding the API port. The frontend container builds the Next.js application in a multi-stage Dockerfile and serves it internally on port~3000, mapped to host port~3001. Redis persists its data in a named Docker volume and is exposed on host port~6380 mapped to container port~6379.

\subsection{Dockerfile and Image Build}

The published images are built from Dockerfiles in \sourcefile{aria-application/}. The backend Dockerfile starts from a Python 3.12 slim image, installs build tools, Ansible, SSH client, network utilities, and the Python dependencies listed in \codepath{requirements.txt}, then copies application code and exposes port~8001. The same image serves both the API and the worker because the Compose file overrides the command at runtime.

The frontend Dockerfile is a multi-stage Node.js 22 Alpine build. It enables \codepath{pnpm}, installs dependencies with \codepath{--frozen-lockfile}, injects the API and WebSocket URLs as build arguments, builds Next.js, and copies only the production artifacts into the final stage. The runner stage exposes port~3000 and starts the application with \codepath{pnpm start}.

Images are built locally with \codepath{docker build}, tagged as \codepath{ghaziiii/aria_project:<service>-latest}, and pushed to Docker Hub. The brain server needs neither source code nor a build environment; only the Compose file, a readable \codepath{.env}, and Docker credentials to pull the images.

\begin{lstlisting}[style=aria,language=bash,caption={Backend Dockerfile excerpt},label={lst:dockerfile-backend}]
FROM python:3.12-slim AS base
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc python3-dev libffi-dev \
    ansible sshpass openssh-client \
    curl ca-certificates
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
ENV PYTHONPATH=/app PYTHONUNBUFFERED=1
EXPOSE 8001
CMD ["python3", "-m", "uvicorn", "api.app:app",
     "--host", "0.0.0.0", "--port", "8001"]
\end{lstlisting}

\begin{lstlisting}[style=aria,language=bash,caption={Frontend Dockerfile multi-stage build excerpt},label={lst:dockerfile-frontend}]
FROM node:22-alpine AS base
ENV PNPM_HOME="/pnpm" PATH="$PNPM_HOME:$PATH"
RUN corepack enable && corepack prepare pnpm@latest --activate

FROM base AS deps
WORKDIR /app
COPY package.json pnpm-lock.yaml .npmrc ./
RUN pnpm install --frozen-lockfile --ignore-scripts

FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
ARG NEXT_PUBLIC_API_URL=http://localhost:8001
ARG NEXT_PUBLIC_WS_URL=ws://localhost:8001/ws
RUN pnpm build

FROM base AS runner
WORKDIR /app
ENV NODE_ENV=production PORT=3000 HOSTNAME="0.0.0.0"
COPY --from=builder /app/public ./public
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/package.json ./package.json
RUN pnpm install --prod --frozen-lockfile --ignore-scripts
EXPOSE 3000
CMD ["pnpm", "start"]
\end{lstlisting}

The playbook performs the following steps for each target host:

\begin{enumerate}[leftmargin=1cm]
\item Stops and disables \codepath{unattended-upgrades} to prevent apt lock conflicts.
\item Cleans up any stuck apt/dpkg/debconf processes and locks.
\item Determines a safe VM name from the remote hostname.
\item Copies \sourcefile{bootstrap_monitored_vm.sh} to \codepath{/tmp} on the remote host.
\item Runs the script in non-interactive mode with the central Elasticsearch IP, credentials, Wazuh group, and environment tag.
\item Streams the script output back to the Ansible console for auditing.
\end{enumerate}

Listing~\ref{lst:ansible-playbook} shows the core tasks of the playbook.

\end{lstlisting}

The operator only needs to keep the Ansible inventory file up to date and run the playbook; every new VM receives the same agents, central target, and index-naming convention. After the playbook finishes, the administrator registers each generated asset payload in ARIA and runs source validation as with manual onboarding.

\section{Production Deployment Workflow}

Putting the whole solution into production follows the same order as the report itself: the order of dependencies. The cloud infrastructure and hardened central host are provisioned first; the monitoring foundation is installed and verified to deliver data into Elasticsearch; the ARIA backend and dashboard are deployed on top; monitored servers are onboarded one by one with the bootstrap script and registered as assets; each asset is source-validated; and only then is remediation selectively enabled per asset. This staged rollout means the platform is never trusted with action before proving it can see data correctly. Figure~\ref{fig:seq-rollout} shows the deployment steps as a sequence: setting the masked \codepath{.env}, running the SQLite migrations, enabling the backend and worker processes under \codepath{systemd}, building and starting the dashboard, running health checks, and finally applying production hardening.

\section{Production Limitations and Hardening}

The platform was validated for an internal, trusted-network deployment---a scope that must be stated honestly. Several items must be addressed before public exposure. There is no full public authentication layer yet: the platform relies on internal trusted-network deployment and the admin-secret gate for dangerous endpoints; role-based access control is not implemented. SQLite suits the current single-node SOC, but a higher-concurrency deployment should strengthen the persistence layer. TLS and certificate handling must be adapted to the production environment rather than bypassed. Finally, Elasticsearch and Redis must remain internal, administrative access should be restricted to trusted networks or a VPN, and only required ports should be exposed. These constraints define the production-hardening work and align with the future work in the general conclusion.

\section{Conclusion}

This chapter completed the engineering story: a single bootstrap script turns any Linux server into a consistently configured, fully attributable monitored asset, ARIA registers and source-validates that asset, and the whole platform is deployed as separated services with a staged, gated production rollout. The solution therefore scales from one self-monitored host to a fleet of servers without losing the separation and control that make it safe. The honest production limitations---authentication, RBAC, persistence, and exposure---frame the remaining work, which the general conclusion now summarizes alongside the project's overall results.
